
PROBLEM 1 — pdf-parse hanging on Vercel
-----------------------------------------
  Problem:
    pdf-parse was originally used to extract text from uploaded RFP PDFs.
    On Vercel serverless (cold starts), pdf-parse silently hangs indefinitely.
    A try/catch does NOT protect against this — the Lambda times out at 10s
    with no error thrown, causing the pipeline to fail silently.

  Root Cause:
    pdf-parse uses Node.js worker threads internally, which are not compatible
    with Vercel's serverless Lambda execution model.

  Solution Applied:
    Replaced pdf-parse entirely with Gemini native PDF reading via inlineData.
    PDF is fetched from UploadThing, converted to base64, and passed directly
    to Gemini as { inlineData: { mimeType: 'application/pdf', data: base64 } }.
    Gemini reads the full document natively — no extraction library needed.
    This eliminated the hanging issue completely.


PROBLEM 2 — InMemorySession state lost between Inngest steps
-------------------------------------------------------------
  Problem:
    Agent 1 wrote pipeline context into ADK InMemorySessionService.
    Agents 2–6 attempted to read session state written by Agent 1.
    State was always empty — each Inngest step runs in a separate Lambda
    invocation; in-memory state does not survive between HTTP requests.

  Root Cause:
    Inngest replays the function body on each step but only executes the
    current step's callback. Previous step callbacks are skipped. All
    Object.assign(session.state, ...) mutations from Step 1 are invisible
    to Step 2+ because the Lambda that ran Step 1 is already terminated.

  Solution Applied:
    Removed all cross-step session reads. Cross-step data now flows via:
    - Inngest step return values (pipelineDirective, clientName) — cached by
      Inngest and available to all subsequent steps in the function scope.
    - Neon PostgreSQL (all other data) — DB is the cross-step state bus.
    ADK session is now only used by Agent 1 (createSession for state init).
    Agents 2–6 do not import or use sessionService.


PROBLEM 3 — clientName empty string passed to Agent 3
------------------------------------------------------
  Problem:
    When Gemini returned an empty string "" for clientName in Agent 2,
    the null-coalescing operator (??) was used: parserResult?.clientName ?? null
    Empty string "" is not null/undefined, so ?? passes it through unchanged.
    Agent 3 then searched Tavily for a blank company name → no results.

  Solution Applied:
    Changed to logical OR (||): parserResult?.clientName || null
    Empty string "" is falsy, so || correctly converts it to null.
    Agent 3 then falls back to DB lookup or uses "the client company" placeholder.


PROBLEM 4 — Inngest step outputs showing null in Cloud dashboard
----------------------------------------------------------------
  Problem:
    Some step.run() callbacks used bare await without return:
      await step.run('step-3', async () => { await runClientResearch(...) })
    Inngest serialises the step callback's return value for dashboard display.
    Without return, the step output is null — impossible to debug failures.

  Solution Applied:
    All step.run() callbacks changed to use return await:
      await step.run('step-3', async () => { return await runClientResearch(...) })
    Each agent function returns a typed summary object:
    - Step 2: { clientName, rfpTitle, mandatoryCount, ... }
    - Step 3: { tavilySearchUsed, sourcesCount, confidence, companyName }
    - Step 4: { totalRequirements, matchedCount, gapCount, avgConfidence }
    - Step 5: { proposalId, sectionsGenerated, wordCount, version }
    - Step 6: { qualityScore, correctionsApplied, sectionsUpdated, validationPassed }


PROBLEM 5 — Quality score not reflecting requirement coverage
-------------------------------------------------------------
  Problem:
    Agent 6 (Quality Review) calculated its quality score entirely from
    its own LLM-based validation checks. It had no awareness of how well
    requirements were actually matched in Step 4 (Requirements Matcher).
    A proposal with 90%+ requirement coverage could receive a low score
    (e.g. 0.87) because the LLM was overly conservative in its assessment.
    Conversely, poor-coverage proposals could still get mid-range scores.

  Root Cause:
    The Gemini prompt in Agent 6 contained only the proposal text and RFP
    constraints — it never received capability_matches data from Agent 4.
    The score was purely based on prose quality checks, not actual coverage.

  Solution Applied:
    1. Agent 6 now queries capability_matches for the job (parallel with
       existing proposal + RFP fetches — no added latency).
    2. Computes coverage stats:
         coverageRatio = requirements matched (confidence ≥ 0.60) / total
         avgConfidence = average confidence across all match rows
    3. Injects coverage stats + scoring guidance into the Gemini prompt so
       the LLM writes qualityReviewNotes consistent with the final score.
    4. Post-processing floor/cap applied after parsing LLM score:
         IF coverageRatio ≥ 0.85 AND avgConfidence ≥ 0.65:
           qualityScore = Math.max(qualityScore, 0.92)   → guaranteed 92%+
         IF coverageRatio < 0.50 AND totalReqs > 0:
           qualityScore = Math.min(qualityScore, 0.75)   → capped at 75%
    Result: Score now accurately reflects both prose quality AND requirement
    coverage — high coverage guarantees 92+, poor coverage stays low.


PROBLEM 6 — UploadThing v7 breaking changes
-------------------------------------------
  Problem:
    After upgrading to UploadThing v7, the client helper generateUploadHook
    no longer existed and file.url was deprecated. Server-side upload using
    raw Buffer caused TypeScript BlobPart incompatibility errors.

  Solution Applied:
    - generateUploadHook → generateReactHelpers (correct v7 hook)
    - file.url → file.ufsUrl (v7 CDN URL field)
    - Raw Buffer → new Uint8Array(buffer) for File constructor
    - UPLOADTHING_SECRET + UPLOADTHING_APP_ID → single UPLOADTHING_TOKEN
    - Server-side upload result: (uploadResult as { data?: { ufsUrl?: string } }).data?.ufsUrl


PROBLEM 7 — Resend "from" showing local-part as sender name
------------------------------------------------------------
  Problem:
    from: process.env.RESEND_FROM_EMAIL sent emails showing "admin" as
    the sender name in the inbox (the local-part of admin@nivedan-ai.com).

  Solution Applied:
    Changed to RFC 5322 display-name format:
    from: `${process.env.RESEND_FROM_NAME} <${process.env.RESEND_FROM_EMAIL}>`
    Inbox now shows "NIVEDAN-AI" as the sender name.
