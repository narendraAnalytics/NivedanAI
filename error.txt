Since local works and Vercel fails, the problem is likely deployment/runtime rather than your code. Your stack uses UploadThing + Next.js + Vercel for document parsing. 

Check these in order:

### 1. Verify ALL UploadThing environment variables

You mentioned:

```env
UPLOADTHING_TOKEN=xxxx
```

Also ensure if your version requires:

```env
UPLOADTHING_TOKEN=
UPLOADTHING_SECRET=
APP_URL=https://yourdomain.vercel.app
```

Redeploy after adding env vars.

---

### 2. Add callback URL explicitly (very common)

Vercel sometimes fails UploadThing callbacks while localhost simulates them automatically. Set:

```env
UPLOADTHING_CALLBACK_URL=https://yourdomain.vercel.app
```

Example:

```env
UPLOADTHING_CALLBACK_URL=https://nivedan-ai.vercel.app
```

UploadThing callbacks failing in production but working locally is a known issue. ([docs.uploadthing.com][1])

---

### 3. Check Vercel Function logs

Open:

**Vercel → Project → Functions → Logs**

Upload PDF again.

Look for errors like:

```
POST /api/uploadthing 401
callback failed
document parse error
timeout
```

If you see:

```
401 /api/uploadthing
```

callback protection/auth issue. ([GitHub][2])

---

### 4. Check Vercel Deployment Protection

If using Preview deployment:

Vercel can block UploadThing callback requests.

Go:

```
Vercel
→ Project Settings
→ Deployment Protection
```

Disable protection temporarily and test. ([Stack Overflow][3])

---

### 5. Check PDF parser runtime issue

Your Nivedan AI uses document parsing libraries (`pdf-parse`, etc.). 

Vercel serverless may fail if parser crashes.

Wrap parser:

```ts
try {
  const text = await parsePdf(fileUrl)
  console.log("PDF parsed", text.length)
} catch (err) {
  console.error("PDF parse failed", err)
}
```

Then inspect Vercel logs.
